基本使用

webpack是一个静态资源打包工具，它会以一个或多个文件作为打包的入口，将我们整个项目所有文件编译组合成一个或多个文件输出出去。

我们将webpack输出的文件叫做bundle。

基本配置

五大核心概念

  1. entry：指定webpack打包时的入口文件。
  2. output：指定webpack打包后的文件输出到哪里，如何命名等。
  3. loader：webpack本身只能做很少的处理，使用loader对其他资源进行处理，如css、字体图标等。
  4. plugins：对webpack的功能进行扩展。
  5. mode：即生产模式和开发模式。

处理js资源

webpack 对 js 处理是有限的，只能编译 js 中 ES 模块化语法，不能编译其他语法，导致 js 不能在 IE 等浏览器运行，所以我们希望做一些兼容性处理(babel)。我们还需要对代码的格式(eslint)进行处理。

Eslint

  1. 简单介绍

    是可组装的Javascript和JSX检查工具。使用Eslint要写Eslint配置文件。

    配置文件的写法很多(有一个即可)：

    • .eslintrc
    • .eslintrc.js
    • .eslintrc.json

    它们的区别在于配置和格式不同，此外还有package.json中的eslintConfig，这种写法不需要创建文件，只需要在原有的文件基础上写。

  2. 配置文件

    接下来依次解释每个配置项：

    • parserOptions

    • rules

      • "off" 或 0 - 关闭规则
      • "warn" 或 1 - 开启规则，使用警告级别的错误：warn (不会导致程序退出)
      • "error" 或 2 - 开启规则，使用错误级别的错误：error (当被触发的时候，程序会退出)

    • extends 继承

      继承现有的规则，就不需要自己写了：

      • Eslint 官方的规则open in new window：eslint:recommended
      • Vue Cli 官方的规则open in new window：plugin:vue/essential
      • React Cli 官方的规则open in new window：react-app

  3. 在webpack中使用

    需要下载eslint-webpack-plugin和eslint。插件(plugin)需要引入才能使用。

    此外，还需要写好eslint的配置文件，新建.eslintrc.js文件，写入相关的配置，打包后就会生效。

SourceMap

我们打包好的main.js代码是复杂的，如果出现了错误，控制台提示的是打包后的文件错误的地方，会使我们很难定位，这时候就要使用SourceMap技术。

下图展示了打包后文件的样子：

下图展示报错时的样子：

SourceMap(源代码映射)是一个用来生成源代码与构建后代码一一映射的文件的方案。它会生成一个 xxx.map 文件，里面包含源代码和构建后代码每一行、每一列的映射关系。当构建后代码出错了，会通过 xxx.map 文件，从构建后代码出错位置找到映射后源代码出错位置，从而让浏览器提示源代码文件出错位置，帮助我们更快的找到错误根源。

webpack devtool 文档告诉我们怎么生成和使用sourcemap。实际开发中，我们常用两种：

  • 开发模式：cheap-module-source-map
    • 打包编译速度快，只包含行映射
    • 没有列映射
  • 生产模式：source-map
    • 包含行/列映射
    • 打包编译速度慢

提升打包构建速度

HotModuleReplacement

有的时候我们只更改了一个文件，重新打包时webpack会把所有模块都进行重新打包编译，速度很慢。我们需要让webpack只重新打包编译更改的文件，这时候需要使用HotModuleReplacement(HMR)。

Thead

当项目越来越庞大时，打包速度越来越慢，甚至于需要一个下午才能打包出来代码。这个速度是比较慢的。我们想要继续提升打包速度，其实就是要提升 js 的打包速度，因为其他文件都比较少。而对 js 文件处理主要就是 eslint 、babel、Terser 三个工具，所以我们要提升它们的运行速度。我们可以开启多进程同时处理 js 文件，这样速度就比之前的单进程打包更快了。

需要注意：请仅在特别耗时的操作中使用，因为每个进程启动就有大约为 600ms 左右开销。
